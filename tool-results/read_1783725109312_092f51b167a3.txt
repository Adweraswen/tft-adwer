   105→## 5. The Live API Reader (`tft_capture/live_api.py`) — THE ROOT OF THE USER'S COMPLAINT
   106→
   107→**Endpoint**: `https://127.0.0.1:2999/liveclientdata/allgamedata`
   108→- Self-signed HTTPS cert → `requests.Session()` with `verify=False`, `urllib3.disable_warnings(InsecureRequestWarning)`.
   109→- No API key required; only works while a TFT game is actually running.
   110→- Poll timeout: 0.8 s. Failure → returns `None` (overlay shows "● maçta değil").
   111→
   112→**The critical code (lines 108–141) — exactly why "only level was being read from the API"**:
   113→
   114→```python
   115→# ─── GOLD — TFT Live API does NOT expose real gold ───────────────────
   116→# The "currentGold" field in TFT is actually total damage dealt (a LoL
   117→# leftover), not your gold. We mark gold as unknown (None) so the
   118→# overlay can show "--" and the screen-capture reader can fill it.
   119→gold = None  # TFT Live API doesn't provide real gold
   120→state.gold = gold if gold is not None else 0
   121→state._gold_from_api = gold is not None
   122→
   123→# ─── LEVEL — activePlayer.level works reliably ───────────────────────
   124→level = self._extract_int(raw, [("activePlayer", "level")])
   125→if level is None and me:
   126→    level = self._extract_int(me, [("level",), ("player", "level"), ("stats", "level")])
   127→state.level = level if level is not None else 1
   128→
   129→# ─── HP — TFT Live API does NOT expose HP ────────────────────────────
   130→# allPlayers[].health is not present in TFT (only in LoL).
   131→# The "isDead" flag tells us if player is eliminated, but not HP value.
   132→hp = None
   133→if me:
   134→    is_dead = me.get("isDead", False)
   135→    if is_dead:
   136→        hp = 0
   137→state.hp = hp if hp is not None else 100
   138→state._hp_from_api = hp is not None
   139→```
   140→
   141→So **the API reliably returns ONLY level** (and "game is running"). HP defaults to 100, gold defaults to 0. This is a fundamental Riot limitation, NOT a bug in this code — TFT's Live API simply does not expose HP/gold/shop/board/augments (those are LoL-only fields). The author discovered this at commit `aa165c99`.
   142→
   143→**Other Live API logic worth knowing**:
   144→- Player lookup: prefers `isCurrentPlayer=True`, falls back to `summonerName == activePlayer.summonerName`, finally `allPlayers[0]`.
   145→- Shop parsing tries 3 paths: `activePlayer.shop.champions`, `activePlayer.shopChampions`, `me.shop.champions`. (In practice returns empty for TFT.)
   146→- Board/bench units: tries `me.board`, `me.units`, `me.champions`, `me.bench`. (Empty for TFT.)
   147→- Augments: scans `events.Events[]` for EventType containing "Augment", plus `me.augments` / `me.runeList` / `me.selectedAugments` fallbacks. (Mostly empty for TFT.)
   148→- Stage/round: tries `gameData.gameStage` / `gameData.round` / top-level fields, then falls back to estimating from `gameTime` via a hardcoded timing table (Stage 1: 0–3 min, Stage 2: 3–8 min, Stage 3: 8–15 min, Stage 4: 15–22 min, Stage 5+: 22+ min; each round ≈ 45–60 s).
   149→- Champion name cleaning: strips `TFT17_` prefix, splits CamelCase into spaces (`TFT17_MissFortune` → `Miss Fortune`).
   150→- XP table: `{1:2, 2:2, 3:6, 4:10, 5:20, 6:36, 7:56, 8:80, 9:84}`.
   151→- Debug dump: with `--debug-api`, every 5th poll writes raw JSON to `debug_api_response.json`.

   194→### 6.3 Cropping + OCR preprocessing (`_crop_and_read`, lines 188–217)
   195→
   196→```python
   197→def _crop_and_read(self, monitor, region_1080, scale):
   198→    x, y, w, h = region_1080
   199→    sx, sy, sw, sh = int(x*scale), int(y*scale), int(w*scale), int(h*scale)
   200→    left = monitor["left"] + sx
   201→    top  = monitor["top"]  + sy
   202→    grab = {"left": left, "top": top, "width": sw, "height": sh, "mon": 0}
   203→    screenshot = self._mss.grab(grab)
   204→    from PIL import Image
   205→    img = Image.frombytes("RGB", screenshot.size, screenshot.bgra, "raw", "BGRX")
   206→    img = img.resize((img.width*2, img.height*2), Image.NEAREST)   # 2x upscale
   207→    text = pytesseract.image_to_string(
   208→        img,
   209→        config="--psm 7 -c tessedit_char_whitelist=0123456789-",
   210→    ).strip()
   211→    return text if text else None
   212→```
   213→
   214→Preprocessing pipeline: **grab BGRA → convert to RGB → 2× NEAREST upscale → Tesseract `--psm 7` (single line of text) with digit-only whitelist `0123456789-`**. No grayscale conversion, no thresholding, no binarization, no contrast normalization — minimal preprocessing.
   215→
   216→### 6.4 The "multi-candidate + range filter" logic (`_read_stat_multi`, lines 219–255) — WHY MISIDENTIFICATION HAPPENS
   217→
   218→```python
   219→def _read_stat_multi(self, monitor, candidates, scale,
   220→                     min_val=0, max_val=9999, stat_name=""):
   221→    best = None
   222→    for i, region in enumerate(candidates):
   223→        text = self._crop_and_read(monitor, region, scale)
   224→        if text:
   225→            match = re.search(r"\d+", text)
   226→            if match:
   227→                val = int(match.group())
   228→                if min_val <= val <= max_val:
   229→                    best = val
   230→                    break     # ← FIRST in-range number wins, even from a wrong region
   231→                # else: out of range, try next candidate
   232→        # Save debug image of this region
   233→        if self._debug and self._mss:
   234→            ... img.save(f"debug_region_{stat_name}_{i}.png")
   235→    return best
   236→```
   237→
   238→**Why this misidentifies regions as gold/HP (exactly the user's complaint)**:
   239→1. The candidate coordinates are guesses. If the actual TFT UI on the user's screen has HP somewhere else (e.g. slightly different x/y because of borderless vs windowed, or Double Up vs normal mode), the "HP candidate" region may actually contain a different UI element (player portrait number, augment icon, timer, etc.).
   240→2. Tesseract is run with a digit-only whitelist, so ANY visual feature that even slightly resembles digits will be read as a number.
   241→3. The function takes the **first** candidate whose OCR result parses to a number in the valid range:
   242→   - HP range: 1–100 (TFT health)
   243→   - Gold range: 0–999
   244→   - Level range: 1–9
   245→4. These ranges are far too permissive to detect a wrong-region misread. If a top-right "HP candidate" actually contains a timer reading "1:23" → OCR returns "123" → 123 > 100 so it'd be rejected (good), but if it contains a player level badge showing "8" → OCR returns "8" → 8 is in [1,100] → accepted as HP=8 (wrong!).
   246→5. Same for gold: if the bottom-center region contains any UI text or icon with digit-like pixels in range 0–999, it's accepted as gold.
   247→6. The "range filter" added in the latest commit (`836fb46f`) was the author's band-aid to reject obvious garbage, but it does NOT verify the region actually contains the intended stat — it just rejects out-of-range OCR output.
   248→
   249→### 6.5 Stage parsing (`_read_stage_multi`, lines 257–273)
   250→
   251→```python
   252→for region in candidates:
   253→    text = self._crop_and_read(monitor, region, scale)
   254→    if text:
   255→        match = re.search(r"(\d+)\s*-\s*(\d+)", text)
   256→        if match:
   257→            stage = int(match.group(1)); rnd = int(match.group(2))

   276→## 7. The Combined Reader (`tft_capture/combined.py`) — How API + OCR Merge
   277→
   278→```python
   279→def poll(self):
   280→    api_state = self._live.poll()              # 1. Live API: detects game, gets level
   281→    if api_state is None:                       # no game running
   282→        self._connected = False
   283→        return None
   284→
   285→    if self._screen is not None:
   286→        screen_state = self._screen.poll()      # 2. Screen OCR: HP/gold/stage
   287→        if screen_state is not None and screen_state.connected:
   288→            api_state.hp    = screen_state.hp
   289→            api_state.gold  = screen_state.gold
   290→            if screen_state.stage > 1 or screen_state.round > 1:
   291→                api_state.stage = screen_state.stage
   292→                api_state.round = screen_state.round
   293→            api_state._hp_from_api    = False
   294→            api_state._gold_from_api  = False
   295→            api_state.source = "live_api"        # ← keep source as live_api so overlay shows "● CANLI"
   296→        # else: screen OCR failed → keep API defaults (hp=100, gold=0)
   297→
   298→    self._connected = True
   299→    return api_state
   300→```
   301→
   302→**Important consequence**: `Overlay._tick()` only treats the state as "in game" if `state.source == "live_api" AND state.connected`. The combined reader hardcodes `source="live_api"` even after merging screen data, so this check passes — but it ALSO means the overlay will show "● CANLI" with bogus HP=100/gold=0 if OCR silently fails. The README explicitly documents this failure mode: "Tesseract yoksa HP=100, gold=0 gösterir" (without Tesseract, shows HP=100, gold=0).

   304→## 8. The Advisor Engine (`advisor/engine.py`) — The Brain
   305→
   306→`AdvisorEngine.recommend(state) -> FullRecommendation` is **stateless per call** (only `_last_comp` and `_committed` persist for pivot tracking). It produces 12 recommendation sub-objects:
   307→
   308→### 8.1 Economy decision (`_economy`, lines 334–369) — gold/econ math
   309→
   310→```python
   311→interest = (state.gold // 10) * 10                          # TFT interest: 1g per 10g, capped at 5g/50g
   312→next_threshold = min(50, interest + 10) if state.gold < 50 else 50
   313→gold_to_next = max(0, next_threshold - state.gold) if state.gold < 50 else 0
   314→
   315→# Target level by stage:
   316→if state.stage <= 2: target_level = 6
   317→elif state.stage == 3: target_level = 7
   318→elif state.stage == 4: target_level = 8
   319→else: target_level = 9
   320→
   321→# Priority decision tree:
   322→if state.hp <= 25:                       return EconomyRec("reroll",   "CRITICAL HP — roll down NOW", ...)
   323→if state.hp <= 40 and state.gold >= 30:  return EconomyRec("reroll",   "Low HP — roll for upgrades", ...)
   324→if state.streak >= 3:                    return EconomyRec("maintain", "Win streak — don't break it", ...)
   325→if state.streak <= -3 and state.stage <= 3:
   326→                                         return EconomyRec("maintain", "Loss streak — keep losing for econ", ...)
   327→if state.level < target_level and state.gold >= 50:
   328→                                         return EconomyRec("level",    "Push to level N — gold at cap", ...)
   329→if state.gold < 50:                      return EconomyRec("save",     "Save toward 50g", ...)
   330→return EconomyRec("maintain", "Strong economy at 50g. Roll above 50.", ...)
   331→```
   332→
   333→Standard TFT economy logic: interest capped at 5 gold (50g banked), reroll thresholds at HP 25/40, target levels 6/7/8/9 by stage.
   334→
   335→### 8.2 Comp planning (`_plan_comp`, lines 225–288)
   336→
   337→- **Early game (stage ≤ 2)**: NO commit — `_early_direction` picks the top 2 active traits and finds a comp that overlaps at least 1 key_trait, labels it "(direction)", confidence 30, strategy "Early game — keep your best board. Don't commit yet."
   338→- **Mid+ (stage ≥ 3)**: scores each of the 12 meta comps:
   339→  - `+12` per core champion on board
   340→  - `+10 * count` per board champion with a key_trait
   341→  - tier bonus: S=+5, A=+2, B=0
   342→  - win_rate bonus: `+0.3 * (win_rate - 50)`
   343→- Picks the highest-scoring comp; if `best_score < 10` → returns None (no comp yet).
   344→- Tracks pivots: if `_last_comp` differs from the new best, sets `pivot_from` field.
   345→
   346→### 8.3 Shop buy/skip (`_shop`, lines 373–423)
   347→
   348→For each of 5 shop slots, decision priority:
   349→1. Is it a 3-star target and you already have ≥1 copy? → **BUY** ("push for 3-star")
   350→2. Is it a core comp unit and you have ≥1 copy? → **BUY**
   351→3. Is it a core comp unit (0 copies) and `gold ≥ cost + 20`? → **BUY**
   352→4. Already have ≥2 copies? → **BUY** ("pushing for 2-star")
   353→5. Trait match and cost ≤ 3? → **BUY**
   354→6. Cost ≥ 4 and `gold ≥ cost + 20`? → **BUY** ("premium standalone")
   355→7. Win-streak early strategy AND cost ≤ 2 AND trait match? → **BUY**
   356→8. Else → **SKIP**
   357→
   358→### 8.4 Reroll decision (`_reroll`, lines 427–434)
   359→
   360→```python
   361→if state.gold < 4:                   return {"should": False, "Not enough gold"}
   362→if econ.action == "reroll":          return {"should": True,  econ.reason}
   363→if state.gold >= 52:                 return {"should": True,  "Above 50g — safe to reroll"}
   364→return {"should": False, "Stay above 50g for max interest"}

   461→## 12. What Was Broken / Limited (matches user's complaint)
   462→
   463→### 12.1 HP and gold CANNOT come from the TFT Live API (root cause)
   464→Riot's TFT Live API (`localhost:2999/liveclientdata/allgamedata`) does not expose HP, gold, shop, board, or augments — those are LoL-only fields. The TFT `currentGold` field is actually "total damage dealt" (a LoL leftover). This is a fundamental Riot limitation, documented in `live_api.py` lines 108–141 and discovered by the author at commit `aa165c99`. **Hence OCR is the only path to real HP/gold — and OCR is broken.**
   465→
   466→### 12.2 OCR misidentifies regions as gold/HP (the user's exact complaint)
   467→The hardcoded `REGIONS_1080p` coordinates in `screen_capture.py` are the author's unverified guesses for 1920×1080. The `_read_stat_multi` function takes the FIRST candidate whose OCR output parses as an in-range number, but:
   468→- The ranges are too permissive (HP 1–100, gold 0–999) to reject wrong-region misreads.
   469→- No preprocessing beyond 2× NEAREST upscale (no grayscale, threshold, binarization, or color filtering) — Tesseract often misreads icons/portrait art/timers as digits.
   470→- The candidate coordinates are not calibrated against the user's actual TFT layout.
   471→- Result: regions meant for HP often capture non-HP UI elements that OCR happens to read as a number in [1,100], and similarly for gold. This is precisely "OCR was misidentifying regions as gold/HP".
   472→
   473→### 12.3 Silent default-to-100/0 failure mode
   474→When OCR fails entirely, `_read_stat_multi` returns None for HP and gold, the `CombinedReader` keeps the API's defaults (`hp=100`, `gold=0`), and the overlay still shows "● CANLI". So the user sees HP=100 / gold=0 with no warning that OCR failed. README documents this: "Tesseract yoksa HP=100, gold=0 gösterir".
   475→
   476→### 12.4 Shop OCR never implemented
   477→`SHOP_SLOT_*` constants are defined (5 slots starting at x=700, y=920, width 100 each) but the comment explicitly says "not currently used (OCR too complex)". The shop array is always `["","","","",""]` from the screen reader. In practice, TFT API also doesn't return shop, so shop-based recommendations (`_shop`, `_reroll`) operate on an empty shop and produce no useful buy/skip advice in live mode.
   478→
   479→### 12.5 Streak always 0
   480→Neither the API nor OCR reads streak. `state.streak = 0` always. This means the economy decision tree's win/loss-streak branches (lines 356–361) never fire — those code paths are effectively dead.
   481→
   482→### 12.6 Pool tracker is fake
   483→`_pool` uses a deterministic PRNG to SIMULATE opponent consumption — not based on real game data. The "contested" warnings in the POOL panel are fabricated.
   484→
   485→### 12.7 Augment parsing is fragile
   486→The 3-path augment parser in `live_api.py` (events with "Augment" EventType, then `me.augments`/`runeList`/`selectedAugments`, then `activePlayer.augments`) mostly returns empty for TFT because the TFT Live API simply doesn't expose augments reliably. So the AUGMENTS panel usually shows "Augment seçimi aktif değil".
   487→
   488→### 12.8 Stale build artifacts bloat the repo
   489→`build/TFTSynapse/` (64 MB `.pkg` + 940 KB `xref-*.html` + ~5 MB `PYZ-00.pyz` + .toc files) was committed despite `.gitignore` listing `build/`. These use the OLD project name "TFTSynapse" (renamed to "TFTAdwer" at commit `bbe0ff36`). This is why GitHub's language stats wrongly report "TeX" and "HTML" as the primary languages instead of Python.

   602→## 14. Rebuild Recommendations (for the next agent)
   603→
   604→Based on the analysis, if rebuilding this project from scratch:
   605→
   606→1. **Keep the architecture as-is** — `tft_capture/` (readers) + `advisor/` (engine) + `data/` (static tables) + `overlay.py` (PyQt5 UI) is a clean separation.
   607→2. **For OCR, do NOT use hardcoded 1080p coordinate guesses with a range filter.** Instead:
   608→   - Use template matching (cv2.matchTemplate) on HP/gold digits — TFT uses fixed fonts, so digit templates are stable.
   609→   - OR use color-based segmentation (HP is yellow/white, gold is yellow-orange) to find the actual digit location dynamically each frame, then OCR just those pixels.
   610→   - OR use a small CNN trained on TFT digit crops (most robust, but requires training data).
   611→   - At minimum, add grayscale + Otsu thresholding before Tesseract — current pipeline skips all preprocessing.
   612→3. **Add a region-calibration tool** — let the user click 4 corners (HP, gold, level, stage) once, save to a JSON config, load on startup. This eliminates the hardcoded-coordinate fragility entirely.
   613→4. **Consider switching from Tesseract to EasyOCR or PaddleOCR** — both handle stylized game fonts better than Tesseract with no preprocessing.
   614→5. **For real game-state beyond level**, the only reliable source is the Live API for level + connection. Everything else MUST come from screen reading. Accept this and design the OCR pipeline to be robust.
   615→6. **The pool tracker is fake** — either remove it or label it clearly as an estimate. Don't present simulated data as real.
   616→7. **The streak is always 0** — either remove streak-based economy advice or find a way to OCR the streak indicator.
   617→8. **Clean up the committed `build/TFTSynapse/` artifacts** — they bloat the repo 67× and break GitHub language stats. The `.gitignore` already lists `build/`; just `git rm -r build/` and commit.
   618→9. **The advisor engine logic is sound** — economy decision tree, comp scoring, shop buy/skip, carry targeting, item scoring, stage awareness — all reuse-worthy as-is.
   619→10. **The data tables (Set 17)** are reusable as-is (61 champions, 36 traits, 12 comps, items, augments, pool sizes, shop odds, XP table).
